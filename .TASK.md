# codex-auditor WebUI 重构设计大纲

## Summary

- 重构当前 WebUI：从单层会话改为 `Target -> Session` 两级结构。
- 现有实现仅作参考，不做最小补丁式兼容；围绕 Flask、SQLite、原生前端和 Codex 运行协议重新设计。
- Session 分为两类：manager 自动启动的是 `mining`，用户人为启动的是 `debug`。
- 每个 session 都需要在 Web 端看到类似 ChatGPT 对话的运行过程：用户消息、Codex 可见事件、工具/状态进度、agent 输出结果和最终消息。
- 漏洞列表不再使用 `overall.json`，统一读取目标工作区 `archives/known_findings.md` 的 Markdown 表格。
- 新增漏洞等级编辑弹窗，可把单条发现修改为 `low`、`medium`、`high`，并回写刷新 `known_findings.md`。
- 左侧边栏中不同 target 使用轮换颜色渲染，便于快速区分目标。
- 每个 target 的加号右侧提供齿轮按钮，用于更新 target 补充说明并原子写回工作区 `init.md`。
- 新建目标和更新补充说明两种场景都提供“AI 扩写补充说明”能力。
- TODO：重写 `code_browser`，作为后续独立子任务规划，不混入本轮 WebUI 主重构实现。

## Implementation Workflow

- 实现前先做大致规划，明确每个部分可能需要哪些功能：
  - SQLite schema 与迁移。
  - Markdown 表格解析与原子回写。
  - Codex 进程管理、JSON event 解析、session 绑定与恢复。
  - 前端树状边栏、聊天流、漏洞表和编辑弹窗。
  - 测试、lint、typing。
- 如果某个功能已有稳定通用库可以明显减少代码量，必须先通知用户并等待用户决定是否安装，不要直接自己实现同类复杂功能。
- 可能需要先征询用户的库包括：
  - Markdown 表格解析/修改库，用于稳定处理 `known_findings.md`。
  - 前端 Markdown 渲染或代码高亮库，用于展示 agent 输出。
  - SSE/WebSocket 辅助库，用于实时推送 session 运行事件。
- AI 扩写补充说明属于独立后台运行，不应阻塞页面其他操作；如果计划使用任务队列库或异步框架，需要先告知用户并等待用户决定。
- 如果用户不安装额外库，默认用 Python 标准库和原生浏览器能力实现最小可靠版本。

## Data Model

- `targets`
  - 保存目标名、slug、补充说明、工作区路径、颜色索引、状态、创建/更新时间。
  - `slug` 用于 `/data/workspace/{target_slug}`，必须是安全目录名。
  - `color_index` 从简单颜色列表中轮换分配，也允许后续手动修改。
- `sessions`
  - 归属某个 target。
  - 保存会话名、`session_type`、提示词、Codex session id、状态、错误信息。
  - `session_type` 只允许 `mining` 或 `debug`。
  - `mining`：由新建 target 或 manager 逻辑自动创建并启动。
  - `debug`：由用户在某个 target 下手动创建，用于追问、验证、复现或专项调试。
- `messages`
  - 保存 session 内所有 Web 可见消息。
  - role 包括 `user`、`assistant`、`system`、`event`、`tool`、`judge`。
  - kind 包括 `message`、`event`、`tool_call`、`tool_result`、`status`、`error`。
- `runs`
  - 记录每次 `codex exec`/`resume` 的 prompt、模型、退出码、日志路径、Codex session id 绑定变化。
  - 保存 event log 文件路径，便于 session 页面恢复完整运行过程。
- `settings`
  - 保存主题、选中的 target/session、树展开状态。

## Workspace And Agent Protocol

- 新建目标时准备 `/data/workspace/{target_slug}`。
- 若目录不存在，从 `/data/workspace/templates` 复制初始化。
- 补充说明写入目标工作区 `init.md` 的“补充说明”段落。
- 更新 target 补充说明时同时更新数据库和 `{target_workspace}/init.md`。
- 写回 `init.md` 必须原子化：先写同目录临时文件，再 `rename` 替换，避免 agent 同时读取到半截内容。
- 写回 `init.md` 时只替换“补充说明”段落，不改动模板其他内容。
- 创建 target 后自动创建一个 `mining` session，并使用 manager 的目标初始化/挖掘逻辑启动 agent。
- 用户在 target 下点击加号创建的是 `debug` session。
- 新 session 首次运行使用 `codex exec --json -C {target_workspace}`。
- 已绑定 Codex session id 的 session 使用 `codex exec resume --json {codex_session_id}`。
- 每次运行保存：
  - Codex JSON events 原始日志。
  - Web 可见事件流。
  - 最终 agent 输出。
  - 退出码。
  - 发现到的 Codex session id。
- 新建 session 不复用旧 Codex session；恢复 session 必须使用该 session 保存的 id。
- 运行中禁止删除、重复启动和编辑同一 session 的运行参数。

## Session Output Protocol

- 每个 session 页面都要像聊天一样展示完整运行过程。
- Web 端展示内容来自 Codex CLI 可见 JSON event、stdout/stderr 摘要、工具调用状态和最终 assistant 输出。
- 不要求展示模型隐藏推理；只展示 Codex event 中明确可见的推理摘要、计划、工具调用、工具结果和 agent 消息。
- 运行中前端需要持续刷新或订阅 session event：
  - 最小实现可以轮询 `/api/sessions/{session_id}/messages`。
  - 如果计划使用 SSE/WebSocket 或相关库，需要先告知用户并等待用户决定。
- 事件展示建议：
  - 用户消息靠右。
  - agent 最终输出靠左。
  - 工具调用、状态更新、judge 信息使用较淡样式靠左。
  - 当前运行中的 session 显示“运行中”状态、耗时和停止按钮。
- 页面切换到任意历史 session 时，必须能恢复显示该 session 的历史消息、事件和最终输出。

## Target Note Protocol

- 补充说明存在两个入口：
  - 新建 target 表单。
  - target 行加号右侧的齿轮按钮。
- 齿轮按钮打开“更新补充说明”弹窗，展示当前 target 名称和现有补充说明。
- 新建 target 和更新补充说明的表单都提供三个常规动作：
  - 取消。
  - 确定。
  - AI 扩写补充说明。
- 点击“AI 扩写补充说明”后，不直接覆盖用户输入。
- 后端使用与挖掘漏洞相同的模型和 effort 启动一次独立 Codex 调用。
- AI 扩写 prompt 固定为：

```text
以下内容为用户对{target}漏洞挖掘的补充说明, 补充扩写来更加明确用户的补充说明边界和用户可能遗漏的问题, 不要落地文件
{用户的补充说明}
```

- prompt 后附加用户当前填写的补充说明内容。
- AI 回复展示在补充说明输入框下方的单独只读框内。
- 扩写结果框提供复制按钮，用户自行决定是否复制到补充说明输入框。
- 扩写失败时只在弹窗内显示错误，不影响原输入内容。
- 更新补充说明点“确定”后：
  - 更新 `targets.note`。
  - 原子写回 `{target_workspace}/init.md` 的“补充说明”段落。
  - 不自动重启已有 `mining` session。

## Vulnerability List Protocol

- 读取路径为 `/data/workspace/{target_slug}/archives/known_findings.md`。
- 表格协议固定为：

```markdown
| summary | bug type | security rating | source files |
| --- | --- | --- | --- |
```

- `security rating` 只接受 `low`、`medium`、`high`。
- 其他等级或空值显示为 unknown 样式，但 WebUI 不自动改写。
- 文件不存在或表格为空时显示“暂无漏洞”，不弹错误。
- Markdown 解析失败时显示读取失败，并提供“请求 AI 修复汇总”操作。
- 挖掘 agent 负责向 `archives/known_findings.md` 追加发现；WebUI 负责读取、展示和人工修改等级。

## Vulnerability Edit Protocol

- 漏洞表每行提供“编辑”按钮。
- 点击后弹窗显示 summary、bug type、source files 和等级选择。
- 用户只能修改 `security rating` 为 `low`、`medium`、`high`。
- 保存时后端重新读取 `known_findings.md`，按稳定 row id 或行号定位该行，只改等级单元格，并原子写回文件。
- 保存成功后重新读取并刷新漏洞列表。
- 若文件在弹窗打开后被 agent 追加导致定位冲突，后端拒绝保存并要求前端刷新。
- 如果采用 Markdown 表格解析/修改库可以降低错误率，必须先告知用户并等待用户决定是否安装。

## UI Design

- 整体仍为左侧边栏 + 右侧上下分区。
- 左侧边栏：
  - 可收起树状结构，根节点是 target，叶节点是 session。
  - 顶部提供“新建目标”。
  - 每个 target 右侧提供加号，用于新建该 target 下的 `debug` session。
  - 每个 target 加号右侧提供齿轮按钮，用于更新该 target 的补充说明。
  - 不同 target 使用简单颜色轮换渲染，例如 green、blue、amber、rose、violet、cyan。
  - target 颜色用于左边框、圆点或小色块，不大面积铺色。
  - `mining` 和 `debug` session 使用不同小图标或短标签区分。
- 新建目标表单：
  - 字段为目标和补充说明。
  - 操作为取消、确定、AI 扩写补充说明。
  - AI 扩写结果显示在补充说明输入框下方的独立只读框内，并提供复制按钮。
  - 创建后自动启动默认 `mining` session。
- 更新补充说明弹窗：
  - 从 target 齿轮按钮打开。
  - 字段为补充说明。
  - 操作为取消、确定、AI 扩写补充说明。
  - AI 扩写结果显示在输入框下方的独立只读框内，并提供复制按钮。
- 新建 debug session 表单：
  - 字段为会话名和初始提示词。
  - 创建后可选择立即发送初始提示词启动，或保存为空闲 session。
- 右侧上半页：
  - 显示漏洞表。
  - 等级用颜色区分。
  - 每行提供编辑入口。
- 右侧下半页：
  - 显示当前 session 的聊天与运行事件。
  - 最底部为输入框、发送按钮和停止按钮。
- 主题：
  - 支持亮色和暗色。
  - 亮色主题以阅读模式纸黄色为主。
  - 暗色主题以接近黑色的深灰色为主。
  - 服务端渲染 index 前读取 settings 并注入 body，避免先亮后暗闪烁。

## API Design

- `GET /api/state`
  - 返回 settings、targets 和每个 target 内嵌的 sessions。
- `POST /api/targets`
  - 创建目标、准备工作区、创建默认 `mining` session、启动默认 agent。
- `PATCH /api/targets/{target_id}`
  - 更新目标名、补充说明或颜色索引。
  - 更新补充说明时必须原子写回 `{target_workspace}/init.md`。
- `POST /api/targets/{target_id}/expand-note`
  - 使用与挖掘漏洞相同的模型和 effort 扩写补充说明。
  - 请求体包含当前补充说明文本。
  - 返回 AI 扩写文本，不修改数据库和 `init.md`。
- `POST /api/targets/expand-note`
  - 用于新建 target 前扩写补充说明。
  - 请求体包含 target 名称和当前补充说明文本。
  - 返回 AI 扩写文本，不创建 target。
- `DELETE /api/targets/{target_id}`
  - 仅允许目标下无 running/judging session 时删除。
- `POST /api/targets/{target_id}/sessions`
  - 创建目标下的 `debug` session。
- `GET /api/sessions/{session_id}/messages`
  - 拉取 session 消息、事件和 agent 输出。
- `POST /api/sessions/{session_id}/messages`
  - 发送消息并启动或 resume Codex。
- `POST /api/sessions/{session_id}/stop`
  - 停止当前运行。
- `GET /api/targets/{target_id}/vulnerabilities`
  - 解析 `archives/known_findings.md`。
- `PATCH /api/targets/{target_id}/vulnerabilities/{row_id}`
  - 修改单条漏洞等级并回写 Markdown。
- `POST /api/sessions/{session_id}/repair-vulnerabilities`
  - 请求 AI 修复或刷新 `known_findings.md`。

## Test Plan

- 数据库：
  - 新库初始化生成所有表和默认 settings。
  - 创建 target 会生成 target、默认 `mining` session、system message 和 run 记录。
  - 更新 target 补充说明会同步更新数据库字段。
  - 创建用户 session 时类型为 `debug`。
  - 删除 running/judging session 或 target 被拒绝。
- Target 补充说明：
  - 新建 target 时补充说明写入 `init.md` 对应段落。
  - 齿轮弹窗更新补充说明时原子写回 `init.md`，不破坏其他段落。
  - AI 扩写接口返回文本但不修改原输入、数据库或 `init.md`。
  - AI 扩写失败时表单仍可取消或确定原内容。
- Session 输出：
  - 运行中 Codex event 能被写入 messages 或 event log。
  - 切换到历史 session 后能看到历史运行过程和最终 agent 输出。
  - 首次运行调用 `codex exec` 并绑定 Codex session id。
  - 后续消息调用 `codex exec resume`。
  - 停止运行后状态为 `interrupted`，session 回到 `idle`。
- Markdown 漏洞列表：
  - 能解析标准 `known_findings.md` 表格。
  - 文件缺失、空表格显示“暂无漏洞”。
  - malformed table 显示读取失败。
  - 编辑等级只修改目标行的 `security rating` 单元格。
  - agent 追加导致文件版本变化时，编辑保存被拒绝并提示刷新。
- 前端：
  - 首屏主题无闪烁。
  - target 树展开、选择 session、聊天和漏洞表刷新联动正常。
  - 不同 target 按颜色轮换渲染。
  - target 行同时显示新建 debug session 加号和更新补充说明齿轮。
  - 新建 target 与更新说明弹窗都能展示 AI 扩写结果并复制。
  - `mining` 与 `debug` session 在边栏可区分。
  - 漏洞编辑弹窗可打开、保存、取消，并正确刷新表格。
- 质量门禁：
  - `ruff check`
  - `mypy`
  - Flask API 单元测试覆盖核心 CRUD、运行状态转换和 Markdown 回写。

## Assumptions

- 不迁移旧单层 session 数据；重构后可从空数据库启动。
- 后端继续使用 Flask，前端继续使用原生 HTML/CSS/JS。
- `overall.json`、`OVERALL.md` 和 overall-report-skill 从 WebUI 主流程移除。
- `known_findings.md` 是漏洞列表唯一数据源。
- WebUI 只展示 Codex 可见事件和输出，不展示模型隐藏推理。
- 额外依赖必须先告知用户，由用户决定是否安装。
